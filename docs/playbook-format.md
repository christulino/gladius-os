# Stage Playbook Format Reference

A **stage playbook** is a markdown file with a YAML frontmatter block, attached to a
stage (or, as a type-level default, to a work item type). When a work item enters
that stage, Gladius can run the playbook: it assembles context, calls an AI model,
and writes the model's output back to the work item's journal.

This document is the authoritative reference for the frontmatter format and the
execution lifecycle. It is generated by reading the implementation directly — the
source files are:

- `runtime/stagePlaybooks.js` — `parsePlaybook()`, `validateContextBudget()`
- `runtime/playbookExecutor.js` — `executePlaybookForStageEntry()`
- `runtime/contextAssembler.js` — `assembleContext()`, `formatContextForPrompt()`
- `runtime/orgAiModels.js` — `resolveModelConfig()`
- `runtime/contextTypes.js` — the journal entry-type taxonomy
- `runtime/transitions.js` — the call site that fires execution

If this document and the code ever disagree, the code wins — file an issue.

---

## Anatomy of a playbook

```markdown
---
trigger: on_enter
model: default
context:
  pull: [discovery, acceptance]
  org: [architecture, standards]
  write: [decision, note]
---

Write playbook instructions here in markdown.
The AI agent will receive the above context and follow these instructions.
```

The file is split by `parsePlaybook()` using the regex `/^---\n([\s\S]*?)\n---\n?([\s\S]*)$/`:

- Everything between the two `---` fences is parsed as YAML (`js-yaml`, restricted to
  `yaml.JSON_SCHEMA` — plain JSON-compatible types only, no custom tags).
- Everything after the closing `---` is the **body**, trimmed of leading/trailing
  whitespace. The body is opaque, freeform markdown — it is not parsed further, and is
  passed to the model verbatim as the task instructions.
- If the file doesn't match the `---`-fenced shape at all (no frontmatter), the whole
  file is treated as the body and frontmatter defaults apply.
- If the frontmatter YAML fails to parse, `parsePlaybook()` silently falls back to
  `{ meta: {}, body: content }` — i.e. all frontmatter defaults apply and the entire
  raw file (including the broken frontmatter block) becomes the body. This fails
  **open**, not closed — a YAML typo doesn't block a save, it just gets treated as if
  no frontmatter existed.

## Frontmatter keys

### `trigger`

**Default: `on_enter`.**

The only two values that actually cause the playbook to run are `on_enter` and
`stage_entry` (treated as synonyms). Any other value — including `on_exit` or
`manual` — means the playbook **never executes** via the in-server executor; there
is no exit-trigger or manual-trigger runner today.

This isn't a save-time validation — an invalid trigger value is accepted and stored,
it simply causes `executePlaybookForStageEntry()` to return early every time the
executor is invoked. The trigger only matters at *run* time: execution is always
initiated by the transition engine on stage entry (see [Execution lifecycle](#execution-lifecycle)
below); the frontmatter's `trigger` key is a gate on that fixed entry point, not a
selector for a different entry point.

### `model`

**Default: `'default'`.**

Must exactly match the `name` column of a row in `blueprint.org_ai_models` for the
work item's org (`resolveModelConfig(orgId, modelName)` does an exact, case-sensitive
lookup with `is_active = true`). There is no fuzzy matching and no fallback model.

If no match is found — including if the org simply has no model named `default` and
the playbook omits `model` — the executor does **not** silently drop the run. As of
the current code:

1. A `runtime.playbook_runs` row is inserted with `status='running'` *before* model
   resolution is attempted.
2. Model resolution fails, and that same row is updated to `status='failed'` with an
   error message: `No AI model named "<modelName>" is configured for this org. Add a
   model with that name in Org Center → AI Models.`

So a bad `model` value is visible in the UI via the playbook-run indicator — it is a
loud failure, not a silent one, as long as you know to look at the run history for
that work item.

### `context.pull`

**Default: `[]`.**

A flat array of journal entry type strings to pull from the *current work item's*
journal (`runtime.context_entries`), e.g.:

```yaml
context:
  pull: [discovery, acceptance, decision]
```

Valid entry types (enforced by a DB CHECK constraint when entries are *written*, not
when pulled — an unrecognized type here just matches zero rows): `nfr`, `discovery`,
`acceptance`, `design`, `decision`, `note`, `test-plan`, `playbook`.

`context.pull` also accepts one special sentinel value in the same array:

```yaml
context:
  pull: [discovery, ancestors]
```

`ancestors` is **not** a journal entry type — it's a flag. When present, Gladius
additionally walks the work item's `parent_id` chain (recursive CTE, up to 50 rows,
all types, most-recent-first) and pulls every journal entry from every ancestor. It
does not filter ancestor entries by the other types listed in `pull` — ancestors
contribute all of their own journal entries regardless of type.

Entries pulled this way are included in the prompt tagged `[human]` or `[agent]`
based on their `is_agent` flag, so the model can distinguish authoritative human
input from prior (possibly hallucinated) agent output.

### `context.org`

**Default: `[]`.**

A flat array of org-context type strings to inject from the org's context library
(`blueprint.org_context`), e.g. `[architecture, standards]`. Unlike journal entry
types, org-context types are **not** constrained to a fixed taxonomy — they're
freely named per org (`runtime/orgContext.js`; see also the comment in
`runtime/contextTypes.js`). Whatever type strings the org actually used when writing
org-context entries are the values to reference here.

### `context.write`

**Default: `['note']`.**

The allowlist of journal entry types the model is permitted to write back. The model
is asked to respond with a JSON array of `{ type, title, content }` objects; each
object's `type` is checked against this allowlist (`Set` membership) before being
persisted — anything not on the list is silently discarded (not written, not logged
as an error). Written entries are created with `isAgent: true` and `authorId: null`
via `createContextEntry()`.

Only the eight journal entry types listed under `context.pull` above are meaningful
here (the DB CHECK constraint rejects anything else at write time — that would throw
inside `executePlaybookForStageEntry`'s try/catch, which is swallowed and recorded on
the run as `status='failed'`).

### `context_budget`

**Default: `24000`** (the module constant `MAX_CONTEXT_CHARS` in `contextAssembler.js`).

Optional. Overrides the character budget applied to the assembled context block
(org context + ancestor journal + item journal, concatenated) before it's inserted
into the system prompt. Material beyond the budget is truncated with a visible
`_[context truncated to budget]_` notice rather than silently dropped.

If present, the value **must be a finite positive number** — this is validated at
*save* time (`POST /stages/:stageId/playbook` and `PATCH
/organizations/:orgId/playbooks/:id` both call `validateContextBudget()`), so a
malformed `context_budget` is rejected with a 400 when you try to save the playbook,
not discovered later at run time. (The executor also re-checks validity defensively
via `isValidContextBudget()` before use, since this code path must never throw.)

### `max_tokens`

**Default: `4096`.**

Optional. Passed straight through as `max_tokens` to the Anthropic `messages.create`
call. If the model's response is cut off at this limit (`stop_reason === 'max_tokens'`),
the executor logs a warning and attempts to salvage whatever complete `{type,
title, content}` objects it can from the truncated JSON (see
`runtime/parseAgentEntries.js`) rather than discarding the whole response.

## Keys that look like frontmatter but aren't

- **`execution_owner`** — this is a **column on the `blueprint.stage_playbooks` row**
  (`'in_server'` default, or `'agent'`), set via the playbook API/UI — not a
  frontmatter key. When `execution_owner = 'agent'`, the in-server executor skips the
  playbook entirely (logs and returns) — that playbook is understood to be run by an
  external agent (e.g. over MCP) instead.
- Do not use `on_exit` or `manual` for `trigger` expecting a different execution
  point to fire it — no such execution point exists today. (See `trigger` above.)
- Do not write `context.pull` as a list of objects (e.g. `{type: discovery, traverse:
  ancestors}`). The parser only understands a flat array of strings, with `ancestors`
  as one of the strings in that same array. (Some in-app AI-assist copy describes the
  object form — that copy is aspirational/inaccurate; this document reflects what the
  parser and executor actually do.)

## Execution lifecycle

1. A work item transitions to a new stage. `runtime/transitions.js` commits the
   transition, then — as a fire-and-forget side effect wrapped in
   `.catch(() => {})`, never blocking or failing the transition response — calls
   `executePlaybookForStageEntry(workItemId, toStageId, orgId, workItemTypeId)`.
2. **Lookup:** `getPlaybookForStage()` finds the most specific *active*
   (`is_active = true`) playbook: a playbook scoped directly to `toStageId` wins over
   one scoped to the work item's type as a type-level default (`stage_id IS NULL`).
   If none is found, or it isn't active, execution stops silently — no playbook_runs
   row is written (there's nothing to run).
3. **Execution-owner check:** if the playbook's `execution_owner` is `'agent'`,
   execution stops here (logged, no run record).
4. **Parse:** `parsePlaybook()` splits frontmatter (`meta`) from body.
5. **Trigger check:** if `meta.trigger` (default `on_enter`) isn't `on_enter` or
   `stage_entry`, execution stops here — silently, no run record.
6. **Run record opens:** a `runtime.playbook_runs` row is inserted with
   `status='running'` — from this point on, every remaining failure path updates
   this same row rather than failing silently.
7. **Model resolution:** `resolveModelConfig(orgId, meta.model ?? 'default')` looks up
   and decrypts the org's AI model config. Failure → run marked `failed`, stop.
8. **Context assembly:** `assembleContext()` pulls item-journal entries
   (`context.pull`), ancestor entries (if `ancestors` is in `context.pull`), and
   org-context entries (`context.org`); `formatContextForPrompt()` renders them into
   one text block, capped at `context_budget` (or the 24,000-char default).
9. **Prompt build:** a system prompt embeds the context block plus the
   `context.write` allowlist and the required JSON-array response shape; the user
   prompt embeds the work item's title/description and the playbook body as
   instructions.
10. **Model call:** `client.messages.create()` via the Anthropic SDK, with a 90s
    timeout and exponential-backoff retry (up to 3 attempts) on HTTP 429/529 only —
    any other error is a hard failure.
11. **Response parsing:** `parseAgentEntries()` tolerantly parses the JSON array,
    salvaging complete objects even if the response was truncated at `max_tokens`.
12. **Write-back:** each parsed entry is checked against the `context.write`
    allowlist and must have non-empty `content`; passing entries are persisted via
    `createContextEntry(workItemId, { type, title, content, isAgent: true, authorId:
    null })`. Entries failing the allowlist check are dropped, not errored.
13. **Run record closes:** updated to `status='success'` with token usage, stop
    reason, and the count of entries written — or `status='failed'` with the error
    message, on any exception in steps 6–12 (the whole function body is wrapped in a
    try/catch that never rethrows).

Playbook runs are visible per work item at `GET
/admin/api/work-items/:id/playbook-runs` and rendered by the `PlaybookRunIndicator`
UI component.

## Complete example

This is a real, parseable playbook — a Planning-stage playbook that pulls prior
discovery notes plus the org's architecture/standards knowledge, and is allowed to
write back a decision and a note. `context_budget` and `max_tokens` are included to
show their syntax, though most playbooks can omit both and take the defaults.

```markdown
---
trigger: on_enter
model: default
context:
  pull: [discovery, acceptance, ancestors]
  org: [architecture, standards]
  write: [decision, note]
context_budget: 16000
max_tokens: 2048
---

You are assisting with the Planning stage of a work item in a Kanban flow system.

Review the discovery notes and any acceptance criteria already captured on this
item and its ancestors, plus the org's architecture and standards context.

Then:
1. If there is an open architectural or scoping question that blocks planning,
   write a `decision` entry framing the question precisely enough for a human to
   answer it. Do not invent an answer.
2. Write a `note` entry summarizing what you found and any gaps in the discovery
   record that a human should fill in before this item leaves Planning.

Do not write acceptance criteria or design entries — those are out of scope for
this stage's playbook.
```

Frontmatter validity check for this example, against `parsePlaybook()` +
`validateContextBudget()`:

- The file matches the `---`-fenced shape, so YAML parsing is attempted.
- `trigger: on_enter` — a recognized value, will actually fire.
- `model: default` — assumes the org has a model literally named `default` in Org
  Center → AI Models; if not, the run will fail loudly (see `model` above).
- `context.pull: [discovery, acceptance, ancestors]` — three strings in a flat
  array; `ancestors` is the sentinel, `discovery`/`acceptance` are valid entry types.
- `context.org: [architecture, standards]` — flat array of freely-named org-context
  types; valid as long as the org actually curates entries under those two names.
- `context.write: [decision, note]` — both valid journal entry types.
- `context_budget: 16000` — a finite positive number, passes `validateContextBudget()`.
- `max_tokens: 2048` — read directly by the executor, no save-time validation applied
  to this key.
